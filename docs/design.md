# ルンバ自動発進システム — 引き継ぎ書 / 構想まとめ

> Dispatch向け。本書をもとにGitHubリポジトリ（README・設計メモ・Node-REDフロー）として構造化することを想定。

最終更新: 2026-05-28

-----

## 1. 目的（結論）

ルンバが充電ステーション（ドック）から**自動で発進できない**問題を、
「発進時刻にドックの通電を約10秒だけ切る」ことで解決する。
制御は手持ちの常時起動サーバ上の **Node-RED** で行い、**技適取得済みのスマートプラグ Tapo P110** をローカル制御する。**追加ハード購入なし**。

-----

## 2. 背景・課題

- ルンバは、発進の瞬間にドックが通電していると発進できない個体挙動がある。
- 検証で判明した事実：**発進予定時刻からドック通電を約10秒オフにすると正常に発進できる**。
- 市販の汎用IoTタップはスケジュール分解能が**1分単位**しかない。
- 通電オフを1分続けると、ルンバがドック（IRビーコン）を見失い、発進地点を把握できず**迷子**になる。
- したがって **オフ時間を「約10秒」に厳密制御** することが要件。

-----

## 3. 確定した解決方針

### 3.1 役割分担

- **タイミング/ロジック（9:00トリガ + 10秒待ち）** → Node-RED が担当。
- **物理的な通電ON/OFF** → スマートプラグ（Tapo P110）が担当。
- 10秒のオフ時間は**プラグ側タイマー（分単位）ではなくNode-RED側で秒精度で握る**。

### 3.2 採用ハード：Tapo P110（無印）

- 国内正規流通品＝**技適取得済み**で合法に使用可。
- **AC100V ネイティブ**・直差しプラグ（Aタイプ）。負荷上限15A（ドック用途は数十W級で余裕）。
- **電力モニタリング機能あり**（後述の事後検証に使用）。
- ローカル制御の実績あり（クラウド非依存運用が可能）。

### 3.3 制御基盤：Node-RED on `tailscale-gateway`

- ホスト：`tailscale-gateway`（Raspberry Pi OS、**常時起動・家庭LAN内**）。
- Node-RED 用 非公式ノード **`node-red-contrib-tapo-new-api`**（KLAPプロトコル対応）でTapoをLAN内制御。
- 運用時の通電制御指令はLAN内で完結＝クラウド非依存。ただしKLAP認証にTP-Linkアカウント情報が必要（後述）。

-----

## 4. 却下した代替案と理由（設計判断の記録）

|案                           |却下理由                                             |
|----------------------------|-------------------------------------------------|
|汎用IoTタップ（手持ち）               |スケジュールが分単位のみ。10秒オフ不可。                            |
|Shelly Plug S / Plus Plug US|技適未取得の懸念（電波法）。EU版はプラグ形状非適合、US版は選定必要。             |
|Shelly 1 Mini               |直差し型ではなく**AC直結リレー**。100Vの自作配線が必要（安全・PSE・電気工事の負担）。|
|プラグ単体運用（Shellyの秒オートオン等）     |動作は可能だが技適等の論点が残る。Node-RED路線なら秒制御は自前で握れるため不要。     |

採用：**Tapo P110 + Node-RED ローカル制御**（技適・100V・形状・秒制御・監視のすべてを満たす）。

## 5. 重要な技術的制約・判明事項

### 5.1 「発進検知→即ON」の閉ループは作れない（要注意）

- 構成は 壁→P110→ドック。**通電OFF中はP110のリレーが開放**され、負荷側電流は常に0。
- 発進には無通電が必須なので、「発進した瞬間」は必ず無通電と重なり、**プラグの電流では原理的に検知不能**。
- よって **ON契機は固定ディレイ（実証済みの約10秒）** とする。リアルタイム検知はしない。

### 5.2 電力モニタリングの使いどころ＝「事後検証」

- ON復帰の数秒後に電流を読み、**発進の成否を事後判定**する補助に使う。
  - 復帰後に充電電流が立つ → ルンバがまだ居る＝**発進失敗**の疑い → 通知。
  - 電流が低いまま → ドックを離れた＝**成功**の傍証。
- **限界**：朝は一晩充電後で満充電に近く、居ても充電電流がほぼ0のことがある。その場合「居る/居ない」の判別が鈍り、事後検証の信頼性は低下する。
- 確実な成功判定が必要なら、電流ではなく **iRobotのAPI/アプリ状態** 等の別信号が要る（負債増。今回はスコープ外）。

### 5.3 その他の運用上の注意

- **初期セットアップのみスマホ必須**（Tapoアプリでの初回Wi-Fi登録）。以降はNode-REDで自走。
- **Tapo IPはDHCP予約で固定**（ノードはIP直指定のため、変動すると停止）。
- **Pi ⇔ Tapo は同一セグメントで相互到達**できること（AP/クライアントアイソレーション無効、VLAN分離時はルーティング許可）。
- KLAPは非公式実装。**FW更新でプロトコルが変わると壊れる可能性**、稀に排他制御エラー/一時的無応答の報告あり → `catch`ノードで失敗を拾い通知し、**無言停止を防ぐ**。
- 型番注意：**無印P110（電力モニタリング）と P110M（Matter対応）は別物**。Node-REDでのエネルギー取得実績は無印P110が確実。Matter不要なら無印を選ぶ。

## 7. 実装手順（フェーズ）

### Phase 1 — Node-RED導入（Raspberry Pi OS）

```bash
bash <(curl -sL https://raw.githubusercontent.com/node-red/linux-installers/master/deb/update-nodejs-and-nodered)
sudo systemctl enable nodered.service
sudo systemctl start nodered.service
```

- `http://<piのIP>:1880` で確認。

### Phase 2 — Tapoノード導入

- パレットの管理 → `node-red-contrib-tapo-new-api` を検索・インストール。

### Phase 3 — フロー構築

- 基本：`inject(毎日09:00)` → `power-off` → `delay 10s` → `power-on`
- 監視付き：上記の後段に `(数秒wait)` → `status` → `電流による事後検証` → `失敗濃厚時のみ通知` を追加。
- 各ノード共通設定：TP-Linkアカウントのメール/パスワード、Tapoの固定IP。

### Phase 4 — 検証（本番投入前）

1. injectを手動実行し「OFF → 約10秒 → ON」を実機確認。
1. 「ルンバ発進 → 迷子にならない」を1サイクル確認。
1. 問題なければ本番時刻で常駐（systemd常駐済み）。

- ※Node-REDフローにユニットテストは非該当。代替として**手動inject検証＋catch監視**で品質担保。

-----

## 8. 受け入れ条件

- 毎日9:00に自動でドックが約10秒だけ無通電になる。
- ルンバが発進し、迷子にならずに清掃を開始する。
- KLAPエラー等で停止した場合、通知が飛ぶ（無言停止しない）。

## 付録：用語・前提

- **KLAP**：Tapo新ファームのローカル認証プロトコル。非公式ノードが対応。
- **事後検証**：発進そのものはリアルタイム検知できないため、ON復帰後の電流で成否を推定する補助的チェック。
- 本構成の通電制御指令はLAN内完結だが、KLAP認証のためTP-Linkアカウントの設定が必要（完全オフライン化はスコープ外）。
